## 4.2 Manejo de Errores

### ✅ Error Boundary Global — Implementado
`app/global-error.tsx` existe con integración Sentry:
- Captura excepciones con `Sentry.captureException(error)`
- UI clara con botón "Intentar de nuevo"
- Renderiza HTML minimal (no depende de layout que podría haber fallado)

### ✅ Logging Estructurado — Bien implementado
`lib/logger.ts` proporciona:
- `devLog()` — solo en desarrollo
- `devWarn()` — dev: console.warn, prod: JSON estructurado
- `devError()` — dev: console.error, prod: Sentry + JSON estructurado
- `logAudit()` — audit trail fire-and-forget en Supabase
- **No se loguean datos sensibles** (notas médicas se anonimizan antes de pasar a IA)

### ✅ Server Actions con `{ success, error }` Estandarizado
Las ~25 server actions retornan consistentemente `{ success: boolean, error?: string }` o `{ exito: boolean, error?: string }`.

### 🟠 Inconsistencia en Patrón de Auth entre Actions (Alta)
Se usan **dos patrones distintos** para obtener el contexto del usuario:

| Patrón | Módulos que lo usan |
|--------|---------------------|
| `getContextoOrg()` / `getOrgIdActual()` | `facturas`, `clasificacion`, `billing`, `equipo`, `dian`, `perfil`, `tarifas`, `radicacion`, `pacientes`, `glosas`, `respuesta-glosas` |
| `supabase.auth.getUser()` directo | `conciliacion`, `benchmarks`, `acuerdos`, `dashboard`, `busqueda-codigos` |

Los módulos que usan `getUser()` directo **no validan la organización** del usuario, lo que significa que consultan por `user_id` en vez de `organizacion_id`. Esto funciona para usuarios individuales pero rompe el modelo multi-tenant para equipos.

### 🔴 Errores Silenciados (Crítica)
3 instancias de `.catch(() => {})` que tragan errores completamente:
- `app/pagos/page.tsx:51` — `listarEPSUsuario().then(setEpsList).catch(() => {})`
- `components/Sidebar.tsx:42` — `obtenerBadgesSidebar().then(setBadges).catch(() => {})`
- `components/Sidebar.tsx:51` — `...catch(() => {})`

Si estas funciones fallan (ej: error de red, token expirado), el usuario no recibe ninguna indicación. Los badges y la lista de EPS simplemente no se cargan sin explicación.

### 🟡 Sin `error.tsx` por Ruta (Media)
No existen archivos `error.tsx` en ningún segmento de ruta. Solo existe `global-error.tsx`. Esto significa que cualquier error en un segmento de ruta (ej: `/facturas`, `/glosas`) muestra el error global en vez de uno contextual que preserve el layout circundante.

### 🟡 Sin `not-found.tsx` (Media)
No existe un `not-found.tsx` custom. URLs inexistentes renderizan la página 404 genérica de Next.js.

### 🟡 Sin `loading.tsx` (Media)
No existen archivos `loading.tsx` en ningún segmento. No hay Suspense boundaries por ruta.

## 4.3 Performance (Análisis Estático)

### 🔴 Llamadas a Gemini AI sin Timeout (Crítica)
Las llamadas a los modelos Gemini (`getMedibillAI()`, `getRagExtractorAI()`, `getHelperAI()`, `getGlosaAI()`) **no tienen timeout configurado**.

En `lib/gemini.ts`, los 3 modelos se crean sin `AbortSignal.timeout()` ni `requestOptions.timeout`. Si Gemini se cuelga o responde lentamente, el server action se bloquea indefinidamente.

Contraste con `lib/providers/muv-client.ts` que sí implementa timeouts:
```typescript
const MUV_TIMEOUT_MS = 30_000;
signal: AbortSignal.timeout(MUV_TIMEOUT_MS)
Impacto: Una clasificación médica o sugerencia de respuesta de glosa podría bloquear un serverless function hasta su timeout de plataforma (~60s en Vercel), consumiendo recursos.

🔴 N+1 Query en Cron de Eliminación de Cuentas (Crítica)
En app/api/cron/eliminar-cuentas/route.ts:56:

for (const uid of userIds) {
  await supabase.auth.admin.deleteUser(uid);
}
Cada usuario se elimina con una llamada individual dentro de un loop. Si hay 100 cuentas pendientes de eliminación, son 100 llamadas API secuenciales.

Nota: supabase.auth.admin.deleteUser no soporta batch nativo, pero se podría paralelizar con Promise.allSettled().

🟠 Queries sin Paginación en Dashboard (Alta)
app/actions/dashboard.ts ejecuta múltiples queries que traen todos los registros sin limit:

Línea 24: facturas.select("valor_total") — todas las facturas del mes
Línea 42: glosas_recibidas.select("valor_glosado") — todas las glosas activas
Línea 52: respuestas_glosas.select("estado") — todas las respuestas históricas
Para usuarios con alta volumetría, estas queries traerán miles de registros para hacer un reduce() en JS. Estos cálculos deberían ser RPCs de PostgreSQL con SUM() / COUNT().

🟠 Queries sin Paginación en Benchmarks (Alta)
app/actions/benchmarks.ts trae facturas, glosas y pagos sin limit para comparativas de EPS:

Línea 87: todas las facturas del usuario por EPS
Línea 115: todos los pagos de esas facturas
Otras queries similares en obtenerResumenBenchmarks() (líneas 208-280)
🟡 Sin Lazy Loading de Componentes (Media)
No se encontró uso de next/dynamic ni React.lazy en ningún componente. Componentes pesados como modales o gráficos se cargan en el bundle principal.

🟡 Sin Uso de next/image en Web App (Media)
La app web (apps/web) no usa next/image. Solo apps/docs lo importa. Si hay imágenes en la app web, no se están optimizando (formatos modernos, lazy loading nativo, responsive sizing).

✅ Rate Limiting Implementado
Redis (Upstash) en producción con sliding window
Fallback in-memory en desarrollo
Aplicado en webhook de Telegram (5 req/min)
✅ Chunking en Queries Largas
app/actions/pagos.ts:34 chunkea queries por factura IDs:
for (let i = 0; i < facturaIds.length; i += CHUNK_SIZE)
✅ listarFacturas Paginada
La lista principal de facturas sí implementa paginación con .range() y devuelve count.

4.4 Patrones y Consistencia
🔴 Duplicación Significativa: Pipeline de Clasificación IA (Crítica)
Archivos afectados:

app/actions/clasificacion.ts — Pipeline web (funciones: extraerTerminosRAG, clasificarConGemini, validarYEnriquecerResultado)
app/actions/telegram-clasificacion.ts — Pipeline Telegram
El pipeline de clasificación IA está duplicado casi línea por línea en telegram-clasificacion.ts:

RAG extraction (parsing de términos, búsqueda de candidatos) — ~30 líneas idénticas
Clasificación con Gemini — idéntico
Validación y enriquecimiento (CUPS, CIE-10, coherencia anatómica, dedup consulta, corrección de diagnostico_asociado, default tipo_servicio) — ~50 líneas idénticas
La versión de Telegram además crea su propia instancia de GoogleGenerativeAI en vez de usar getGenAI() de lib/gemini.ts.

Riesgo: Un fix o mejora en el pipeline web no se propaga al pipeline Telegram. Ya hay divergencias menores (Telegram no aplica tarifas del usuario, no registra auditoría).

🟠 Inconsistencia en Nombres de Propiedades de Retorno (Alta)
Las server actions usan dos patrones distintos para el campo de éxito:
Patrón	Módulos
{ success: true/false }	facturas, billing, equipo, cuenta, dian, conciliacion, admin
{ exito: true/false }	clasificacion, acuerdos, telegram-clasificacion
Esto obliga a los consumers a conocer cuál patrón usa cada action.
