### Servicios Externos Integrados

1. **Supabase** — Base de datos PostgreSQL, Auth, Storage (soportes-glosas bucket), RLS
2. **Google Gemini AI** — Clasificación médica (CIE-10/CUPS), sugerencias de respuesta a glosas, mapper de sábanas EPS
3. **Matias API** — Facturación electrónica DIAN (envío, consulta estado, descarga XML/PDF firmados)
4. **MUV (MinSalud)** — Mecanismo Único de Validación para RIPS → CUV (proxy en Cloud Run)
5. **Wompi** — Pasarela de pagos Colombia (PSE, Nequi, tarjetas)
6. **Resend** — Envío de emails transaccionales (radicación)
7. **Upstash Redis** — Rate limiting distribuido
8. **Sentry** — Monitoreo de errores en producción
9. **Telegram Bot API** — Bot para clasificación por audio

## 0.4 — MAPA DE ROLES Y PERMISOS

### Modelo de Roles

El sistema tiene **dos niveles** de roles:

1. **Roles de organización** (RBAC): `owner`, `admin`, `doctor`, `facturador`, `auditor`
2. **Platform admin**: Super-admin global (tabla separada `platform_admins`)

### Jerarquía de Roles
platform_admin (super-admin global, tabla separada)
└── owner (dueño de la organización)
└── admin (administrador de la organización)
├── doctor (médico)
├── facturador (personal de facturación)
└── auditor (solo lectura + responder glosas)

### Flujo de Autenticación y Middleware
Request → Middleware
│
├── /api/* → PASS (API routes manejan auth internamente)
│
├── No user + no /login + no /invitacion → REDIRECT /login
│
├── User con email NO confirmado → signOut + REDIRECT /login?email_no_confirmado
│
├── User en /login → REDIRECT /dashboard
│
├── User sin onboarding (cookie miss) → CHECK DB perfiles.onboarding_completo
│ ├── No completo → REDIRECT /onboarding
│ └── Completo → SET cookie + PASS
│
├── User en /admin/* → CHECK platform_admins table
│ ├── No admin → REDIRECT /dashboard
│ └── Es admin → SET cookie (5min TTL) + PASS
│
├── User con suscripción cancelada/unpaid → REDIRECT /configuracion/suscripcion
│ (Excluye: /login, /onboarding, /api, /configuracion/suscripcion, /invitacion, /admin)
│
└── Todo ok → PASS

### Resumen de Seguridad

| Capa | Implementación |
|---|---|
| Autenticación | Supabase Auth (email/password, email confirmation required) |
| Autorización de rutas | Middleware (Next.js) |
| Autorización de datos | RLS (PostgreSQL) — `auth.uid()` + `get_user_org_id()` |
| Autorización de acciones | RBAC (`lib/permisos.ts`) — `verificarPermisoOError()` en server actions |
| Rate Limiting | Upstash Redis + fallback Supabase RPC (por IP para login, por user para API) |
| CSRF | Next.js built-in (server actions) |
| CSP | Headers en next.config.js (sin unsafe-eval) |
| HSTS | Strict-Transport-Security con preload |
| Frame Protection | X-Frame-Options: DENY + frame-ancestors: 'none' |
| Input Sanitization | Prompt injection filter (AI helper) |
| Encryption at rest | MUV credentials: AES-256-GCM |
| Audit Logging | `audit_log` table para acciones críticas |
| Error Monitoring | Sentry (client + server + edge) |
