5.2 — WEBHOOKS Y CALLBACKS
WEBHOOK 1: Wompi (/api/wompi/webhook)
Archivo: app/api/wompi/webhook/route.ts (L1-100)

Autenticidad
PASS ✅ — Verificación de firma HMAC-SHA256 en wompiProvider.verificarWebhookFirma() (lib/wompi.ts L148-168):
Concatena transaction.id + status + amount_in_cents + timestamp + events_secret
Compara con signature.checksum del payload
Rechaza con 401 si no coincide
Idempotencia
HIGH 🟠 — NO es idempotente. Si Wompi envía el mismo webhook dos veces con transaction.approved:

suscripciones.update se ejecuta de nuevo — sobrescribe fechas pero no duplica filas (OK por ser UPDATE)
historial_pagos.insert SÍ crea un registro duplicado — no hay ON CONFLICT ni verificación de wompi_transaction_id existente
organizaciones.update de wompi_payment_source — idempotente (sobrescribe con mismo valor)
IMPACTO: Registros duplicados en historial_pagos. La suscripción no se duplica, pero el historial de pagos se contamina.

FIX RECOMENDADO: Agregar verificación antes del insert:
// Verificar si ya procesamos esta transacción
const { data: existing } = await supabase
  .from("historial_pagos")
  .select("id")
  .eq("wompi_transaction_id", evento.transactionId)
  .single();
if (existing) break; // Ya procesado
O agregar UNIQUE constraint en historial_pagos(wompi_transaction_id).

Tiempo de respuesta
PASS ✅ — El webhook procesa todo sincrónicamente pero solo hace 2-3 queries a Supabase (update + insert). Debería completar en <1 segundo. No hay procesamiento pesado delegado.
WEBHOOK 2: Telegram (/api/telegram/webhook)
Archivo: app/api/telegram/webhook/route.ts (L1-660)

Autenticidad
HIGH 🟠 — Verificación de autenticidad insuficiente. El webhook verifica que TELEGRAM_BOT_TOKEN exista pero no valida que el request proviene de Telegram:
// L43-46: Solo verifica que el token exista en env vars, NO que la request sea de Telegram
const botToken = process.env.TELEGRAM_BOT_TOKEN;
if (!botToken) {
  return NextResponse.json({ error: "Bot no configurado" }, { status: 500 });
}
Telegram recomienda usar secret_token al registrar el webhook con setWebhook() y verificar el header X-Telegram-Bot-Api-Secret-Token en cada request. Esto NO está implementado.

IMPACTO: Cualquiera que descubra la URL /api/telegram/webhook puede enviar updates falsos. Un atacante podría:

Simular mensajes de usuarios que no existen
Consumir clasificaciones de Gemini sin autorización
Agotar el rate limit de otros usuarios
FIX RECOMENDADO:

Agregar TELEGRAM_WEBHOOK_SECRET env var
Pasar secret_token al llamar setWebhook
Verificar request.headers.get("x-telegram-bot-api-secret-token") en L42
Idempotencia
WARN ⚠️ — Parcialmente idempotente. Si Telegram re-envía un update:

Clasificaciones: se ejecutarían de nuevo (duplicado en auditorías_rips y uso_mensual incrementado doble). Sin embargo, Telegram raramente re-envía si el webhook retorna 200 (que siempre hace, L38).
Vinculación: el código de vinculación ya está usado=true, así que el re-envío retorna "ya fue usado" (OK).
MITIGANTE: El webhook siempre retorna 200, por lo que Telegram no re-envía. Riesgo real bajo.

Tiempo de respuesta
HIGH 🟠 — Puede exceder 5 segundos fácilmente. El procesamiento de audio en handleAudio() hace:

obtenerArchivo() → fetch a Telegram API (~200ms)
descargarArchivo() → fetch a Telegram API (~500ms-2s para audio 5MB)
clasificarAudioTelegram() → 3 llamadas a Gemini (transcripción 3s + RAG ~2s + clasificación ~3s) = **8-12 segundos**
Validaciones DB + enviar respuesta (~500ms)
Total estimado: 12-16 segundos para audio processing. Telegram espera respuesta en ~60 segundos, así que no timeout, pero Vercel serverless (10s free, 60s Pro) podría ser el bottleneck.

FIX RECOMENDADO: Para deployment Free tier de Vercel (<10s timeout), considerar:

Responder 200 inmediatamente, procesar en background
O asegurar plan Pro de Vercel (60s timeout)
