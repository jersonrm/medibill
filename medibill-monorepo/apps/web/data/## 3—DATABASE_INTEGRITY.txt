## 3.1 Schema y Constraints

### 3.1.1 Columnas NOT NULL — ¿default o requeridas en lógica de inserción?

| Tabla.Columna | NOT NULL | Default | ¿Cubierta en código? | Veredicto |
|---|---|---|---|---|
| `audit_log.action` | YES | `''` (string vacío) | La función `logAudit()` en logger.ts siempre envía `action` | ⚠️ DEFAULT vacío es inútil — debería ser NOT NULL sin default |
| `audit_log.accion` | YES | ninguno | `registrarAuditLog()` siempre envía `accion` | ✅ OK — requerido en código |
| `audit_log.user_id` | YES | ninguno | Código verifica `if (!user) return` antes de insertar | ✅ OK |
| `facturas.valor_total` | YES | `0` | Código siempre envía `valor_total` desde input | ✅ OK |
| `facturas.valor_glosado` | YES | `0` | Valor por defecto correcto | ✅ OK |
| `facturas.valor_aceptado` | YES | `0` | Valor por defecto correcto | ✅ OK |
| `organizaciones.email_billing` | YES | ninguno | `crearOrganizacion()` siempre envía `emailBilling` | ✅ OK |
| `perfiles.numero_documento` | YES | ninguno | Requerido en onboarding | ✅ OK |
| `perfiles.codigo_habilitacion` | YES | ninguno | Requerido en onboarding | ✅ OK |

**Hallazgos críticos:**

1. **🔴 `audit_log` tiene columnas duplicadas: `accion` (TEXT NOT NULL) y `action` (TEXT NOT NULL DEFAULT '').** Dos scripts crearon la misma tabla con esquemas distintos: `schema-audit-log.sql` usa `accion`, y `001-rate-limits-and-audit.sql` usa `action`. Ambas columnas coexisten en producción. El código usa ambas: `registrarAuditLog()` escribe en `accion`, mientras `logAudit()` escribe en `action`. Resultado: toda fila insertada por un camino tendrá la otra columna vacía/null.

2. **⚠️ `acuerdo_tarifas.created_at` permite NULL** — no tiene NOT NULL. Lo mismo para `acuerdos_voluntades.created_at`, `auditorias_rips.creado_en`, `facturas.created_at`, y muchas más. Aunque el default `now()` lo cubre, una inserción explícita con `NULL` lo aceptará.

## 3.2 Consistencia de Datos

### 3.2.1 Registros huérfanos — queries diagnósticas

```sql
-- Facturas sin user_id válido en auth.users
SELECT f.id, f.num_factura, f.user_id 
FROM facturas f 
LEFT JOIN auth.users u ON u.id = f.user_id 
WHERE u.id IS NULL;
-- RESULTADO ESPERADO: 0 filas (solo 4 usuarios y 9 facturas)

-- Facturas con paciente_id inválido
SELECT f.id, f.num_factura, f.paciente_id 
FROM facturas f 
WHERE f.paciente_id IS NOT NULL 
AND NOT EXISTS (SELECT 1 FROM pacientes p WHERE p.id = f.paciente_id);
-- SIN FK CONSTRAINT — podrían existir huérfanos

-- Facturas con resolucion_id inválido (sí tiene FK)
-- Protegido por FK constraint — OK

-- glosas_recibidas sin factura válida
SELECT gr.id, gr.factura_id 
FROM glosas_recibidas gr 
WHERE NOT EXISTS (SELECT 1 FROM facturas f WHERE f.id = gr.factura_id);
-- Protegido por FK — OK

-- acuerdo_tarifas sin acuerdo válido
-- Protegido por FK con ON DELETE CASCADE — OK

-- acuerdos_voluntades.prestador_id → sin FK constraint

FROM acuerdos_voluntades av 
WHERE NOT EXISTS (SELECT 1 FROM perfiles p WHERE p.id = av.prestador_id);
-- ⚠️ SIN FK — potenciales huérfanos

3.2.2 Duplicados donde no debería haberlos
-- facturas.num_factura — TIENE UNIQUE constraint ✅
-- Sin embargo, borradores usan "BORR-{timestamp}" que se reemplaza al aprobar.
-- Si el proceso de aprobación falla entre generar número y update, podría
-- haber duplicados de BORR- (no crítico).

-- perfiles con mismo numero_documento (diferentes orgs)
SELECT numero_documento, COUNT(*) 
FROM perfiles 
GROUP BY numero_documento 
HAVING COUNT(*) > 1;
-- ⚠️ NO HAY UNIQUE en perfiles.numero_documento — un mismo NIT podría tener
-- múltiples perfiles si un usuario se registra dos veces

-- organizaciones con mismo NIT
SELECT nit, COUNT(*) 
FROM organizaciones 
WHERE nit IS NOT NULL 
GROUP BY nit 
HAVING COUNT(*) > 1;
-- ⚠️ NO HAY UNIQUE en organizaciones.nit — dos orgs con mismo NIT es posible

-- suscripciones — protegido por UNIQUE(organizacion_id) ✅

-- usuarios_organizacion — protegido por UNIQUE(organizacion_id, user_id) ✅

⚠️ perfiles.numero_documento no tiene UNIQUE constraint a nivel global (solo compuesto perfiles_user_id_key en user_id).

3.3 RLS y Políticas de Acceso
3.3.1 ¿RLS habilitado en TODAS las tablas con datos de usuario?
Tabla	RLS	Datos sensibles	Veredicto
catalogo_causales_glosa	OFF	No — catálogo público read-only	✅ OK — datos de referencia
cups_alias	OFF	No — alias de códigos CUPS	⚠️ Debería tener RLS con policy SELECT para authenticated, o al menos proteger INSERT/UPDATE/DELETE
cups_categorias	OFF	No — categorías de códigos	⚠️ Mismo que cups_alias
reglas_coherencia	OFF	No — reglas de validación	⚠️ Pero permite INSERT/UPDATE/DELETE sin restricción
Todas las tablas con datos de usuarios tienen RLS=ON. ✅

Pero hay un problema grave con permisos:

3.3.2 🔴 CRÍTICO: Permisos del rol anon
TODAS las tablas otorgan permisos completos (SELECT, INSERT, UPDATE, DELETE, TRUNCATE, TRIGGER, REFERENCES) al rol anon. Esto significa que cualquier persona sin autenticación, usando solo la anon key pública, podría ejecutar operaciones si no hubiera RLS.

Para tablas CON RLS habilitado, las policies protegen (la mayoría usa auth.uid() que retorna NULL para anon). Pero:

catalogo_causales_glosa (RLS OFF): anon tiene DELETE, INSERT, UPDATE. Un atacante con la anon key podría borrar o modificar todo el catálogo de causales de glosa (215 registros de referencia normativa).

cups_alias (RLS OFF): anon puede INSERT/UPDATE/DELETE. Podría inyectar alias maliciosos en las búsquedas CUPS.

cups_categorias (RLS OFF): anon puede INSERT/UPDATE/DELETE.

reglas_coherencia (RLS OFF): anon puede INSERT/UPDATE/DELETE. Podría desactivar reglas de coherencia médica.

rate_limits (RLS ON, sin policies): RLS sin policies = bloqueo total para anon/authenticated. Solo funciona vía check_rate_limit() SECURITY DEFINER. ✅ Correcto.

Tablas con RLS ON pero policy USING=true (lectura abierta):

acuerdo_tarifas: SELECT USING=true — cualquier authenticated puede ver TODAS las tarifas pactadas de TODOS los prestadores 🔴
acuerdos_voluntades: SELECT USING tiene OR true al final — misma vulnerabilidad 🔴
auditoria_plazos: "Lectura publica plazos" USING=true — todos los plazos de todos los usuarios visibles ⚠️

Resumen de Hallazgos — Clasificación por Severidad
🔴 CRÍTICOS (requieren acción inmediata)
Tablas sin RLS expuestas al rol anon con permisos de escritura:

catalogo_causales_glosa, cups_alias, cups_categorias, reglas_coherencia — un atacante con la anon key pública puede DELETE/INSERT/UPDATE datos de referencia.
acuerdo_tarifas y acuerdos_voluntades tienen SELECT USING=true:

Cualquier usuario autenticado puede ver las tarifas pactadas y acuerdos de TODOS los prestadores. Información comercial sensible expuesta.
audit_log INSERT policy permite insertar con user_id ajeno:

Policy auth.uid() IS NOT NULL permite que user A inserte un log atribuido a user B.
facturas.user_id no tiene FK a auth.users:

Si se elimina un usuario, sus facturas quedan huérfanas e inaccesibles.
Columnas duplicadas en audit_log (accion + action):

Dos code paths insertan en columnas diferentes, ninguna auditoría está completa.

SECURITY DEFINER sin SET search_path en 5 funciones críticas (get_user_org_id, user_has_role, etc.).

organizaciones.tipo CHECK incluye 'ips' pero TypeScript no — datos inconsistentes code↔DB.

facturas.organizacion_id permite NULL — multi-tenant incompleto.

Sin sistema de migraciones versionado — scripts sueltos, estado DB diverge del código.

Sin CHECK para valores monetarios no-negativos en facturas, glosas.

auditoria_plazos tiene policy SELECT USING=true que overridea la policy restricta.

  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM acuerdos_voluntades av
      WHERE av.id = acuerdo_tarifas.acuerdo_id
      AND av.organizacion_id = get_user_org_id()
    )
  );

  FOR SELECT USING (organizacion_id = get_user_org_id());

  FOR INSERT WITH CHECK (auth.uid() = user_id);
