## 6.2 Observabilidad

### ✅ ¿Existe logging estructurado para producción?

**SÍ**

`lib/logger.ts` implementa un sistema de logging dual:
- **Desarrollo**: `console.log/warn/error` con prefijo `[DEV]`
- **Producción**: JSON estructurado con `{ level, label, message, ts }` + Sentry para errores

Funciones disponibles:
- `devLog(label, ...args)` — solo en desarrollo
- `devWarn(label, ...args)` — JSON en producción
- `devError(label, ...args)` — JSON en producción + `Sentry.captureException`/`captureMessage`
- `logAudit(supabase, event)` — fire-and-forget a tabla `audit_log`

**GAPS:**
- ⚠️ No hay request ID/correlation ID. Un request no se puede trazar end-to-end (ver sección de trazabilidad).
- ⚠️ No hay log level configurable en producción (siempre emite warn y error, nunca info).

---

### ✅ ¿Hay monitoreo de errores configurado (Sentry, etc.)?

**SÍ — Implementación sólida**

- **Client** (`sentry.client.config.ts`): `tracesSampleRate: 0.1`, `replaysOnErrorSampleRate: 1.0`, enabled solo en producción.
- **Server** (`sentry.server.config.ts`): `tracesSampleRate: 0.1`, enabled solo en producción.
- **Edge** (`sentry.edge.config.ts`): Configuración equivalente.
- **Instrumentation** (`instrumentation.ts`): Carga condicional de configs server/edge + `Sentry.captureRequestError`.
- **Global Error Boundary** (`app/global-error.tsx`): Captura errores React no manejados con `Sentry.captureException`.
- **Source Maps** (`next.config.js:55-61`): `withSentryConfig` con `widenClientFileUpload: true`.
- **Logger integration** (`lib/logger.ts:24-26`): `devError` envía a Sentry automáticamente.

**GAPS:**
- ⚠️ DSN hardcodeado (mencionado arriba) — si se compromete, no se puede rotar sin deploy.
- ⚠️ `tracesSampleRate: 0.1` (10%) puede ser insuficiente para debugging en stage/producción temprana. Considerar 1.0 en staging.
- ❌ No se envían breadcrumbs custom de operaciones de negocio (facturación, pagos, MUV).

### ✅ ¿Se pueden trazar requests de un usuario a través de toda la stack?

**NO — GAP SIGNIFICATIVO**

- ❌ No hay `x-request-id`, `traceId`, ni `correlationId` propagado entre componentes.
- ❌ El middleware (`middleware.ts`) no inyecta headers de trazabilidad.
- ❌ Los logs de `lib/logger.ts` no incluyen `userId`, `orgId`, ni `requestId` de forma automática.
- ⚠️ Sentry performance tracing (10%) permite algo de correlación para los requests muestreados, pero no es determinista.
- ✅ `audit_log` tiene `user_id` y `accion`, pero es fire-and-forget y no cubre el flujo completo del request.

**Impacto**: En producción, correlacionar un error con el flujo completo de un usuario requiere buscar manualmente en Sentry + logs de Supabase + audit_log.

### Hallazgos CRITICAL

1. **16 variables de entorno sin documentar en `.env.example`** — incluyendo todas las de pagos (Wompi), email (Resend), encriptación (MUV), y secretos de cron. Un nuevo desarrollador no puede configurar el proyecto.

2. **Sentry DSN hardcodeado** en `sentry.client.config.ts:4`, `sentry.server.config.ts:4`, `sentry.edge.config.ts:4` — debería ser `process.env.SENTRY_DSN` o `process.env.NEXT_PUBLIC_SENTRY_DSN`.

3. **Sin health check en la app principal** — Vercel no necesita uno explícito para deploys, pero no hay endpoint para monitoreo externo (UptimeRobot, Pingdom, etc.).

4. **Sin trazabilidad de requests** — imposible debugging efectivo en producción con múltiples usuarios concurrentes.

5. **Sin plan de disaster recovery** — para una app de facturación médica con datos de salud, esto es un riesgo regulatorio y operativo alto.

### Acciones Recomendadas (por prioridad)

1. **P0**: Completar `.env.example` con las 16 variables faltantes (30 min)
2. **P0**: Mover Sentry DSN a env var (5 min)
3. **P1**: Crear endpoint `GET /api/health` que verifique Supabase + Redis (1h)
4. **P1**: Agregar `x-request-id` en middleware y propagarlo a logger (2h)
5. **P1**: Confirmar plan de Supabase y estado de PITR — documentar en README
6. **P2**: Crear DISASTER_RECOVERY.md con runbooks
7. **P2**: Configurar CD (deploy automático a Vercel desde CI)
8. **P3**: Implementar métricas de negocio (facturas, pagos, errores MUV)

Hallazgo de seguridad relevante durante la auditoría: El archivo .env.local contiene claves reales (Google AI, Supabase service role, Resend, Telegram bot token). Está correctamente en .gitignore tanto en el .gitignore root del monorepo como en el de apps/web/. Sin embargo, los artefactos de build en .next/ (que también están gitignored) contienen estas claves embebidas en los chunks compilados — si alguien accede al filesystem del build, las claves quedan expuestas. Esto es comportamiento normal de Next.js, pero refuerza la importancia de no exponer build artifacts.
