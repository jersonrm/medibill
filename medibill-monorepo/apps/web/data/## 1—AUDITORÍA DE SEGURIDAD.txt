## 1. AUTENTICACIÓN

### 1.1 Manejo de sesiones
**PASS** — Sesiones gestionadas por Supabase Auth vía JWT tokens server-side con `@supabase/ssr`. El middleware (`middleware.ts:38`) llama `supabase.auth.getUser()` que valida el token contra la BD (no solo decode local). Los tokens se almacenan en cookies gestionadas por Supabase SSR con configuración automática de HttpOnly. Supabase maneja refresh tokens internamente.

**Evidencia**: `middleware.ts#L38`, `lib/supabase-server.ts#L10-L30`

### 1.2 Protección contra brute force
**PASS** — Rate limiting implementado en login y registro:
- Login: 5 intentos / 15 min por IP (`login/actions.ts#L9`)
- Registro: 3 intentos / hora por IP (`login/actions.ts#L11`)
- Implementación con Upstash Redis en producción, fallback in-memory en dev (`lib/rate-limit.ts`)

### 1.3 Almacenamiento seguro de tokens
**PASS** — Cookies de Supabase Auth gestionadas por `@supabase/ssr` con configuración server-side. Cookie de onboarding configurada con `httpOnly: true, sameSite: 'lax'` (`middleware.ts#L96-L99`). Cookie de admin con `httpOnly: true, sameSite: 'lax', maxAge: 300` (`middleware.ts#L121-L125`).

1.5 Telegram webhook sin secret token
⚠️ HIGH — TELEGRAM_WEBHOOK_NO_SECRET

Archivo: app/api/telegram/webhook/route.ts#L43-L46

El webhook de Telegram NO valida el X-Telegram-Bot-Api-Secret-Token header. Cualquier persona que conozca la URL del webhook puede enviar updates falsos, potencialmente:

Vincular cuentas de Telegram arbitrarias
Disparar clasificaciones con audio malicioso
Consumir créditos de IA del usuario
Remediación: Configurar secret_token al registrar el webhook con setWebhook, y validar el header X-Telegram-Bot-Api-Secret-Token en cada request.

2. AUTORIZACIÓN
2.1 Verificación de rol por endpoint
🔴 CRITICAL — MISSING_RBAC_IN_CORE_ACTIONS

Las siguientes server actions de alto impacto NO verifican permisos RBAC a pesar de que existe un sistema de permisos definido en lib/permisos.ts:

Action File	Funciones sin RBAC	Permiso Esperado
actions/facturas.ts	crearFacturaBorrador, aprobarFactura, anularFactura, editarFacturaBorrador, marcarComoDescargada	crear_factura, aprobar_factura, anular_factura
actions/dian.ts	Todas las funciones de envío DIAN	enviar_dian
actions/clasificacion.ts	clasificarTextoMedico	clasificar_ia
actions/glosas.ts	Todas las funciones	ver_glosas, responder_glosa
actions/acuerdos.ts	Todas las funciones	gestionar_acuerdos
actions/conciliacion.ts	Todas las funciones	importar_sabana
actions/perfil.ts	guardarPerfil, guardarResolucion	config_organizacion
Solo billing.ts y equipo.ts implementan RBAC.

Impacto: Un usuario con rol auditor (solo lectura) puede crear facturas, aprobarlas, enviarlas a DIAN, modificar el perfil, etc. Un facturador puede clasificar con IA (solo doctor/admin/owner).

Remediación: Agregar verificarPermisoOError(ctx.rol, "accion") al inicio de cada server action que muta datos.

2.2 Escalación de privilegios
PASS — Los admin server actions (admin.ts) verifican correctamente mediante requirePlatformAdmin() que hace un check real contra la BD con service role key, no un simple cookie check.

Nota: El middleware usa una cookie medibill_is_platform_admin como cache (5 min TTL), pero los server actions SIEMPRE re-verifican contra la BD.

2.3 IDOR (Insecure Direct Object References)
PASS — Todas las queries incluyen filtro de ownership:

facturas.ts: .eq("organizacion_id", ctx.orgId) en todas las operaciones
pacientes.ts: .eq("user_id", user.id)
factura-pdf/[id]/route.ts: .eq("id", id).eq("user_id", user.id)
clasificacion-pendiente/route.ts: .eq("token", token).eq("user_id", user.id)
No se encontraron queries que acepten un ID externo sin verificar ownership.

2.4 Middleware de autorización centralizado
⚠️ MEDIUM — INCONSISTENT_AUTH_PATTERN

Existen 3 patrones de autenticación usados inconsistentemente:

supabase.auth.getUser() directo (acuerdos, dashboard, dian, glosas, etc.)
requireUser() helper (cuenta.ts)
getContextoOrg() que incluye rol y org (billing, equipo, facturas, clasificacion)
El patrón 1 NO proporciona acceso al rol del usuario, por lo que las actions que lo usan no pueden hacer RBAC.

Remediación: Migrar TODAS las server actions a usar getContextoOrg() como punto de entrada único de auth + authz.

2.6 Bypass de RLS con service_role key
PASS — El SUPABASE_SERVICE_ROLE_KEY se usa correctamente solo en:

3. VALIDACIÓN DE ENTRADA
3.1 Validación client/server
🔴 CRITICAL — NO_SERVER_SIDE_VALIDATION_LIBRARY

No se utiliza Zod, Yup, joi, ni ninguna librería de validación de schemas en las server actions. La única validación estructurada existente es:

AJV para validar el JSON RIPS de salida (lib/validadorRips.ts)
Validaciones inline básicas (null checks, typeof, .length)
Server actions que aceptan datos complejos sin validación de schema:

Action	Parámetro	Riesgo
crearFacturaBorrador(input: CrearFacturaInput)	Objeto completo con diagnósticos, procedimientos, valores monetarios	Datos corruptos en BD
editarFacturaBorrador(facturaId, datos)	Objeto parcial con datos clínicos	Inyección de campos no esperados
guardarPerfil(datos)	Partial del perfil	Mass assignment potencial
guardarPaciente(datos)	PacienteInput completo	Sin validación de formatos
registrarGlosa(...)	Múltiples parámetros	Sin validación
guardarAcuerdo(...)	Datos de contrato	Sin validación
Impacto: Un atacante puede enviar datos malformados, valores negativos, strings excesivamente largos, o campos inesperados directamente a las server actions.

Remediación: Implementar validación con Zod en cada server action. Ejemplo:
const CrearFacturaSchema = z.object({
  nit_erp: z.string().min(1).max(20),
  valor_total: z.number().nonnegative().max(999999999),
  // ...
});

5.5 Encriptación at rest
WARNING — NO_ENCRYPTION_AT_REST
